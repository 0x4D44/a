# HLD: Saved Exit Codes in Chain Commands

## Problem

Chain commands can only inspect the **previous** command's exit code. There's no way to act on an **earlier** command's result. This blocks patterns like:

```
tollens stop → build → copy → start (only if tollens was running)
```

The exit code from `tollens stop` is lost by the time `tollens start` runs.

## Design

Add two features to chain commands:

1. **`--save <name>`** — a modifier on the *preceding* command that saves its exit code under a named label.
2. **`--if-saved <name> <code> <command>`** — a new chain operator that runs a command only if a previously saved exit code matches.

### Example

```
a add tup "tollens stop" --save was_running \
    --always "cargo build --release" \
    --and "mdrcp" \
    --if-saved was_running 0 "tollens start"
```

Behaviour:
- `tollens stop` runs. Its exit code is saved as `was_running`. (Exit 0 = was running and stopped; exit 2 = wasn't running.)
- `cargo build --release` always runs (regardless of stop's exit code).
- `mdrcp` runs if build succeeded.
- `tollens start` runs only if `was_running == 0`.

## Data Model Changes

### `ChainCommand` — add `save_as`

```rust
struct ChainCommand {
    command: String,
    operator: Option<ChainOperator>,
    #[serde(skip_serializing_if = "Option::is_none")]
    save_as: Option<String>,             // NEW
}
```

`save_as` is orthogonal to `operator` — it's a side effect, not a flow control decision. A command can have both an operator and a `save_as`.

### `ChainOperator` — add `IfSaved`

```rust
enum ChainOperator {
    And,
    Or,
    Always,
    IfCode(i32),
    IfSaved { name: String, code: i32 },  // NEW
}
```

## Execution Changes

### `execute_sequential_chain`

Add a `HashMap<String, i32>` to track saved exit codes. After each command executes, if it has `save_as`, store the exit code. When evaluating `IfSaved`, look up the named value:

```rust
fn execute_sequential_chain(&self, chain: &CommandChain, additional_args: &[String]) -> Result<(), String> {
    let mut last_exit_code = 0;
    let mut saved_codes: HashMap<String, i32> = HashMap::new();  // NEW

    for (index, chain_cmd) in chain.commands.iter().enumerate() {
        let should_execute = match &chain_cmd.operator {
            None => true,
            Some(ChainOperator::And) => last_exit_code == 0,
            Some(ChainOperator::Or) => last_exit_code != 0,
            Some(ChainOperator::Always) => true,
            Some(ChainOperator::IfCode(code)) => last_exit_code == *code,
            Some(ChainOperator::IfSaved { name, code }) => {       // NEW
                saved_codes.get(name).copied() == Some(*code)
            }
        };

        // ... existing skip/execute logic ...

        // After execution, save exit code if requested
        if let Some(ref label) = chain_cmd.save_as {               // NEW
            saved_codes.insert(label.clone(), last_exit_code);
        }
    }
}
```

**Edge case**: If `IfSaved` references a name that was never saved, the command is skipped (the lookup returns `None`, which doesn't match any code). This is intentional — a missing save is treated as "condition not met", not an error.

**Skipped commands**: If a command is skipped (due to its operator), its `save_as` is **not** stored. The saved value reflects actual execution, not hypothetical results.

## CLI Parsing Changes

### `--save <name>`

`--save` is a modifier on the **last added command**, not a new command:

```rust
"--save" => {
    if i + 1 < args.len() {
        if let Some(last_cmd) = commands.last_mut() {
            last_cmd.save_as = Some(args[i + 1].clone());
        }
        i += 2;
    }
}
```

### `--if-saved <name> <code> <command>`

Parsed like `--if-code` but with an extra argument:

```rust
"--if-saved" => {
    if i + 3 < args.len() {
        let name = args[i + 1].clone();
        let code = args[i + 2].parse::<i32>()?;
        commands.push(ChainCommand {
            command: args[i + 3].clone(),
            operator: Some(ChainOperator::IfSaved { name, code }),
            save_as: None,
        });
        i += 4;
    }
}
```

## Display Changes

### `command_display()` (compact, used in `--list`)

Show `--save` inline and `IfSaved` as a new operator symbol:

```
tollens stop @[was_running] ; cargo build --release && mdrcp ?s[was_running=0] tollens start
```

- `@[name]` indicates a saved exit code
- `?s[name=code]` indicates an if-saved check

### Info view (detailed breakdown)

```
Command breakdown:
  1. tollens stop  (saves exit code as 'was_running')
  2. cargo build --release  (always run)
  3. mdrcp  (run if previous succeeded)
  4. tollens start  (run if 'was_running' == 0)
```

### Runtime output

```
[1/4] Executing: tollens stop  [saving exit code as 'was_running']
[2/4](;) Executing: cargo build --release
[3/4](&&) Executing: mdrcp
[4/4](?s[was_running=0]) Executing: tollens start
```

Or when skipped:

```
[4/4] Skipping: tollens start (saved 'was_running' was 2, expected 0)
```

## Help Text Addition

Under the "CHAINING OPERATORS" section:

```
  --save <name>                Save previous command's exit code under <name>
  --if-saved <name> <N> <cmd>  Run if saved exit code <name> equals N
```

## What's NOT Changing

- **Parallel chains**: `--save` and `--if-saved` are sequential-only. Parallel chains ignore both (saves have no defined ordering in parallel). No error — they're just silently skipped.
- **Simple commands**: Unaffected. `save_as: None` is skipped during serialisation.
- **Existing config.json**: Old aliases deserialise fine — `save_as` defaults to `None` when absent, and no existing alias uses `IfSaved`.

## Tollens Side

To make the alias work, `tollens stop` needs to return distinct exit codes:

- **Exit 0**: daemon was running and was stopped successfully
- **Exit 2**: daemon was not running (nothing to stop)

Currently `stop_daemon()` returns `Ok(())` in both cases. The change is small: the CLI handler maps "not running" to `std::process::exit(2)`.
