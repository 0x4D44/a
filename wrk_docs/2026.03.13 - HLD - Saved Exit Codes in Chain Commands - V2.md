# HLD: Saved Exit Codes in Chain Commands (V2)

## Problem

Chain commands can only inspect the **previous** command's exit code. There's no way to act on an **earlier** command's result. This blocks patterns like:

```
tollens stop → build → copy → start (only if tollens was running)
```

The exit code from `tollens stop` is lost by the time `tollens start` runs.

### Prerequisite: Tollens Exit Codes

This feature requires `tollens stop` to return distinct exit codes:

- **Exit 0**: daemon was running and was stopped successfully
- **Exit 2**: daemon was not running (nothing to stop)

Currently `stop_daemon()` returns `Ok(())` in both cases. The tollens CLI handler must map "not running" to `std::process::exit(2)`. **This is a tollens change and must land first.**

## Design

Add two features to chain commands:

1. **`--save <name>`** — a modifier on the *preceding* command that saves its exit code under a named label.
2. **`--if-saved <name>=<code> <command>`** — a new chain operator that runs a command only if a previously saved exit code matches.

### Example

```
a add tup "tollens stop" --save was_running \
    --always "cargo build --release" \
    --and "mdrcp" \
    --if-saved was_running=0 "tollens start"
```

Behaviour:
- `tollens stop` runs. Its exit code is saved as `was_running`.
- `cargo build --release` always runs (regardless of stop's exit code).
- `mdrcp` runs if build succeeded.
- `tollens start` runs only if `was_running == 0`.

## Data Model Changes

### `ChainCommand` — add `save_as`

```rust
#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
struct ChainCommand {
    command: String,
    operator: Option<ChainOperator>,
    #[serde(skip_serializing_if = "Option::is_none")]
    save_as: Option<String>,             // NEW
}
```

`save_as` is orthogonal to `operator` — it's a side effect, not a flow control decision. A command can have both an operator and a `save_as`.

Note: `skip_serializing_if` keeps the serialised JSON clean (no `"save_as": null` on every command). Backwards compatibility with old configs is handled by serde's built-in `Option` deserialization — missing fields default to `None` automatically.

### `ChainOperator` — add `IfSaved`

```rust
#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
enum ChainOperator {
    And,
    Or,
    Always,
    IfCode(i32),
    IfSaved { name: String, code: i32 },  // NEW
}
```

Serde's externally-tagged format serialises this as `{"IfSaved": {"name": "x", "code": 0}}`, consistent with `{"IfCode": 42}` for the existing tuple variant. Old configs (which never contain `IfSaved`) deserialise without issue.

### Single-command chains with `save_as`

Currently, `commands.len() == 1 && !parallel` collapses to `CommandType::Simple(String)`, discarding the `ChainCommand` struct. When `save_as` is set on any command, force the `CommandType::Chain` path to preserve the annotation:

```rust
let has_save = commands.iter().any(|c| c.save_as.is_some());
let command_type = if commands.len() == 1 && !parallel && !has_save {
    CommandType::Simple(commands[0].command.clone())
} else {
    CommandType::Chain(CommandChain { commands, parallel })
};
```

## Execution Changes

### `execute_sequential_chain`

Add a `HashMap<String, i32>` to track saved exit codes:

```rust
fn execute_sequential_chain(&self, chain: &CommandChain, additional_args: &[String]) -> Result<(), String> {
    let mut last_exit_code = 0;
    let mut saved_codes: HashMap<String, i32> = HashMap::new();

    for (index, chain_cmd) in chain.commands.iter().enumerate() {
        let should_execute = match &chain_cmd.operator {
            None => true,
            Some(ChainOperator::And) => last_exit_code == 0,
            Some(ChainOperator::Or) => last_exit_code != 0,
            Some(ChainOperator::Always) => true,
            Some(ChainOperator::IfCode(code)) => last_exit_code == *code,
            Some(ChainOperator::IfSaved { name, code }) => {
                saved_codes.get(name).copied() == Some(*code)
            }
        };

        if !should_execute {
            // ... existing skip logic, plus IfSaved-specific message ...
            continue;
        }

        // ... existing execute logic ...

        // After execution, save exit code if requested
        if let Some(ref label) = chain_cmd.save_as {
            saved_codes.insert(label.clone(), last_exit_code);
        }
    }
}
```

### Semantic rules

1. **Missing save name**: If `IfSaved` references a name that was never saved, the command is skipped. This case is caught at parse time (see validation below), so at runtime it would only occur if a command with `save_as` was itself skipped.

2. **Skipped commands don't save**: If a command is skipped (due to its operator), its `save_as` is **not** stored. Saved values reflect actual execution only.

3. **`last_exit_code` unchanged on skip**: When any command is skipped (including `IfSaved` skips), `last_exit_code` retains the value from the last command that actually ran. This is consistent with how `And`, `Or`, and `IfCode` skips already behave.

4. **Duplicate save names**: Last-write-wins. If two commands save to the same name, the later one overwrites.

## CLI Parsing Changes

### `--save <name>`

`--save` is a modifier on the **last added command**, not a new command:

```rust
"--save" => {
    if i + 1 >= args.len() {
        eprintln!("Error: --save requires a name");
        std::process::exit(1);
    }
    match commands.last_mut() {
        Some(cmd) => {
            cmd.save_as = Some(args[i + 1].clone());
        }
        None => {
            eprintln!("Error: --save must follow a command");
            std::process::exit(1);
        }
    }
    i += 2;
}
```

`--save` is the only backward-looking modifier. All other `--` flags introduce a new command. This is documented in the help text as: "Save the exit code of the preceding step as `<name>`."

### `--if-saved <name>=<code> <command>`

Uses `name=code` format (2 args total, matching `--if-code`'s 2-arg pattern):

```rust
"--if-saved" => {
    if i + 2 >= args.len() {
        eprintln!("Error: --if-saved requires <name>=<code> and a command");
        std::process::exit(1);
    }
    let (name, code) = parse_name_code(&args[i + 1])?;  // splits on '='
    commands.push(ChainCommand {
        command: args[i + 2].clone(),
        operator: Some(ChainOperator::IfSaved { name, code }),
        save_as: None,
    });
    i += 3;
}
```

`parse_name_code` splits `"was_running=0"` into `("was_running", 0)`, erroring if the format is invalid or the code isn't numeric.

### Validation at parse time

After building the command list, validate:

1. **`--if-saved` names reference existing `--save` names**: Every `IfSaved { name, .. }` must have a matching `save_as` somewhere earlier in the chain. Catches typos immediately.

2. **No `--save` or `--if-saved` with `--parallel`**: If `parallel` is true and any command has `save_as` or `IfSaved`, reject with a clear error. Saves have no defined ordering in parallel execution.

```rust
// After building commands, before creating CommandType:
if parallel {
    let has_saves = commands.iter().any(|c| c.save_as.is_some());
    let has_if_saved = commands.iter().any(|c| matches!(c.operator, Some(ChainOperator::IfSaved { .. })));
    if has_saves || has_if_saved {
        eprintln!("Error: --save and --if-saved cannot be used with --parallel");
        std::process::exit(1);
    }
}

let save_names: Vec<&str> = commands.iter()
    .filter_map(|c| c.save_as.as_deref())
    .collect();
for cmd in &commands {
    if let Some(ChainOperator::IfSaved { name, .. }) = &cmd.operator {
        if !save_names.contains(&name.as_str()) {
            eprintln!("Error: --if-saved references '{}' but no prior --save defines it", name);
            std::process::exit(1);
        }
    }
}
```

## Display Changes

### `command_display()` (compact, used in `--list`)

Attach save annotation directly to the command (no space) using `@name`. Use `?s[name=code]` for the IfSaved operator:

```
tollens stop@was_running ; cargo build --release && mdrcp ?s[was_running=0] tollens start
```

- `cmd@name` — this command's exit code is saved as `name`
- `?s[name=code]` — run if saved `name` equals `code`

### Info view (detailed breakdown)

```
Command breakdown:
  1. tollens stop  (saves exit code as 'was_running')
  2. cargo build --release  (always run)
  3. mdrcp  (run if previous succeeded)
  4. tollens start  (run if 'was_running' == 0)
```

### Runtime output

When executing:
```
[1/4] Executing: tollens stop  [saving as 'was_running']
[2/4](;) Executing: cargo build --release
[3/4](&&) Executing: mdrcp
[4/4] Executing: tollens start  (was_running was 0)
```

When skipped:
```
[4/4] Skipping: tollens start (saved 'was_running' was 2, expected 0)
```

When skipped due to missing save (command that saves was itself skipped):
```
[4/4] Skipping: tollens start (saved 'was_running' not set)
```

## Help Text

Under the "CHAINING OPERATORS" section:

```
  --save <name>                    Save the exit code of the preceding step as <name>
  --if-saved <name>=<N> <command>  Run if saved exit code <name> equals N
```

## What's NOT Changing

- **Parallel chains**: `--save` and `--if-saved` are rejected at parse time when combined with `--parallel`.
- **Simple commands**: Unaffected. Single commands without `save_as` still collapse to `CommandType::Simple`.
- **Existing config.json**: Old aliases deserialise fine — `save_as` defaults to `None` when the field is absent (serde's built-in `Option` behaviour).

## Design Boundaries

This feature adds named state to chains. To keep the alias manager simple:
- Saves are scoped to a single chain execution (no persistence between invocations).
- No plans for loops, variables, string interpolation, or other programmability features. If an alias needs more than conditional execution based on exit codes, it should call a script.
